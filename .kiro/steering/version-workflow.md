# 版本管理工作流程

## 版本號規則

### 格式
- 使用格式：`v1.0.2.XXX`
- 每次修改都必須進版
- 版本號必須在所有相關檔案中保持一致

### 版本類型
- **穩定版本**: 經過完整測試，功能正常的版本
- **實驗版本**: 嘗試新功能或修復，可能不穩定
- **回滾版本**: 基於舊版本進行修改的版本

## 進版流程

### 必須更新的檔案
1. 主程式檔案的視窗標題（如 `ui_modern.py`, `ui_modular.py`）
2. 建置腳本的資料夾名稱（如 `build_simple.py`, `build_modular.py`）
3. `CHANGELOG.md` - 新增版本記錄
4. Git commit 訊息

### 標準流程
1. **修改程式碼** - 完成功能開發或問題修復
2. **更新版本號** - 在所有相關檔案中同步更新
3. **更新 CHANGELOG.md** - 記錄詳細的變更內容
4. **測試功能** - 確保修改正常運作
5. **建立執行檔** - 使用對應的建置腳本
6. **Git commit** - 提交變更並推送

## 測試驗證

### 必要測試
- Python 腳本直接執行測試
- 執行檔功能測試
- API 連接測試
- 報告生成測試

### 測試工具
- 使用 `test_modules.py` 進行模組化測試
- 手動測試 UI 功能
- 驗證配置檔案載入和儲存

## 版本追蹤

- **最新版本**: v1.0.2.019 (UI 修正穩定版)
- **穩定基礎**: v1.0.2.009 (已知可用版本)

### 版本決策原則
- 基於穩定版本進行開發
- 避免在問題版本基礎上繼續開發
- 必要時回滾到穩定版本重新開始

## 發布管理

### 建置腳本選擇
- `build_simple.py` - 標準版本建置
- `build_modular.py` - 模組化版本建置
- 根據版本特性選擇對應的建置腳本

### 發布包內容
- 執行檔
- 必要的工具資料夾（fs-reporter, fs-report）
- 配置範本（config.example.txt）
- 文檔（README.md, USAGE_GUIDE.md）
- 使用說明

### 重要提醒
- Python 腳本和執行檔的行為可能不同
- 測試時要同時測試兩種執行方式
- 每次修改都要進版，避免版本混淆
